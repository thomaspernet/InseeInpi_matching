Normalisation du fichier INPI.¶

Le fichier INPI doit contenir un seul fichier gz avant d'tre ingr par le programme. Le fichier va tre import dans un format Dask, ce qui permet de parallliser les calculs et bien sur d'viter les problmes de mmoire.

La normalisation du fichier de l'INPI se fait en plusieurs tapes:

1) Exclusion des observations avec des NaN pour chacune des variables candidates, savoir:

- Adresse_Ligne1
- Adresse_Ligne2
- Adresse_Ligne3
- Code_Postal
- Ville
- Code_Commune

2) Extraction des SIREN a SIRETISER -> cela evite d'utiliser toute la base INSEE pour la sirtisation. I.e Speedup le process 3) Calcul du nombre de SIRET par SIREN via la fonction nombre_siret_siren 4) Normalisation de la variable commune via la fonction clean_commune

- Extraction des digits dans la ville. En effet, certaines communes incluent l'arrondissement dans la variable.
- Extraction des caractres spciaux et espaces
- Capitalisation du nom de la commune
- Matching avec le fichier commune pour avoir le nom de la commune de l'INSEE.

5) Prparation de l'adresse via la fonction prepare_adress

- Concatenation des variables `Adresse_Ligne1` + `Adresse_Ligne2` + `Adresse_Ligne3`
- Normalisation de la variable concatene -> Extraction des caractres speciaux, espace, digit puis capitalisation
- Extraction de tous les stop words du fichier `upper_word`
- Split de chaque mot restant de l'adresse 
- Creation du regex de la forme suivante:  `MOT1$|MOT2$` 
- Extration des digits:
    - Premire variable avec le premier digit
    - Seconde variable avec une liste de digit et jointure -> DIGIT1|DIGIT2
- Merge avec le fichier `voie` pour obtenir le type de voie de l'INSEE
- Calcul du nombre de digit dans l'adresse
    - Si len infrieure a 2, alors NaN. C'est une variable utlise pendant le matching des rgles spciales
- Creation d'une variable `index` correspondant  l'index du dataframe. Indispensable

Le fichier est sauvegard en format gz, et dans un table SQL

- inpi_etb_stock_0.gz
- inpi_origine.db

Algorithme Siretisation¶

Le code source est disponible ici et le notebook pour lancer le programme est disponible ici

L'algorithme de SIRETISATION fonctionne avec l'aide de trois fonctions:

  • step_one: permet d'carter les doublons du merge et d'appliquer les premires rgles afin de connaitre l'origine de la siretisation
  • step_two_assess_test: dtermine l'origine du matching, a savoir la date, adresse, voie, numro de voie
  • step_two_duplication: permet de rcuperer des SIRET sur les doublons manant du merge avec l'INSEE

Dans premier temps, on crer un dictionnaire avec toutes les variables de matching. Toutefois, l'algorithme va utiliser squentiellement les variables suivantes:

 {'ncc', 'Code_Postal', 'Code_Commune', 'INSEE', 'digit_inpi'},
 {'ncc', 'Code_Postal', 'Code_Commune', 'INSEE'},
 {'ncc', 'Code_Postal', 'Code_Commune', 'digit_inpi'},
 {'ncc', 'Code_Postal', 'Code_Commune'},   
 {'ncc', 'Code_Postal'},
 {'ncc'},
 {'Code_Postal'},
 {'Code_Commune'}

Pour connaitre l'ensemble des variables de matching INSEE/INPI, veuillez vous rendre en annexe.

Dans la mesure ou l'algorithme fonctionne de manire squentielle, et utilise comme input un fichier de l'INPI a siretiser. De fait, aprs chaque squence, l'algorithme sauvegarde un fichier gz contenant les siren a trouver. Cette tape de sauvegarde en gz permet de loader le fichier gz en input en Dataframe Dask.

Step One¶

La premire tape de la squence est l'ingestion d'un fichier gz contenant les SIREN a trouver. L'ingestion va se faire en convertissant le dataframe en Dask. L'algorithme tout d'abord utiliser la fonction step_one et produit deux dataframes selon si le matching avec l'INSEE a dbout sur des doublons ou non.

Les doublons sont gnrs si pour un mme nombre de variables de matching, il existe plusieurs possibilits l'INSEE. Par exemple, pour un siren, ville, adressse donne, il y a plusieurs possibilits. Cela constitue un doublon et il sera trait ultrieurement, dans la mesure du possible.

Les tapes droules lors du premier processus est le suivant:

- Test 1: doublon
        - non: Save-> `test_1['not_duplication']`
        - oui:
            - Test 2: Date equal
                - oui:
                    - Test 2 bis: doublon
                        - non: Save-> `test_2_bis['not_duplication']`
                        - oui: Save-> `test_2_bis['duplication']`
                - non:
                    - Test 3: Date sup
                        - oui:
                            - Test 2 bis: doublon
                                - non: Save-> `test_3_oui_bis['not_duplication']`
                                - oui: Save-> `test_3_oui_bis['duplication']`
                        - non: Save-> `test_3_non`

Deux dataframe sont cres, un ne contenant pas de doublon et un deuxime contenant les doublons. L'algorithme va raliser les tests sur le premier et faire d'avantage de recherche sur le second.

step_two_assess_test¶

Le premier dataframe ne contient pas de doublon, il est donc possible de raliser diffrents tests afin de mieux dterminerl'origine du matching. Plus prcisement, si le matching a pu se faire sur la date, l'adresse, la voie, numro de voie et le nombre unique d'index. Les rgles sont dfinies ci-dessous.

- Test 1: address libelle
            - Si mots dans inpi est contenu dans INSEE, True
        - Test 1 bis: address complement
            - Si mots dans inpi est contenu dans INSEE, True
        - Test 2: Date
            - dateCreationEtablissement >= Date_Dbut_Activit OR
            Date_Dbut_Activit = NaN OR (nombre SIREN a l'INSEE = 1 AND nombre
            SIREN des variables de matching = 1), True
        - Test 3: siege
            - Type = ['SEP', 'SIE'] AND siege = true, True
        - Test 4: voie
            - Type voie INPI = Type voie INSEE, True
        - Test 5: numero voie
            - Numero voie INPI = Numero voie INSEE, True

Un premier fichier gz est enregistr contenant les "pure matches"

step_two_duplication¶

Les second dataframe contient les doublons obtenus aprs le matching avec l'INSEE. L'algorithme va travailler sur diffrentes variables de manire squentielle pour tenter de trouver les bons SIRET. Plus prcisment, 3 variables qui ont t rcemment cr sont utilises:

  • test_join_address -> True si la variable test_address_libelle = True (ie mot INPI trouv dans INSEE) et test_join_address = True
  • test_address_libelle -> True si la variable test_address_libelle = True (ie mot INPI trouv dans INSEE)
  • test_address_complement -> True si la variable test_join_address = True

Pour chaque squence, on ralise les tests suivants:

- Si test_join_address = True:
        - Test 1: doublon:
            - Oui: append-> `df_not_duplicate`
            - Non: Pass
            - Exclue les `index` de df_duplication
            - then go next
        - Si test_address_libelle = True:
            - Test 1: doublon:
                - Oui: append-> `df_not_duplicate`
                - Non: Pass
                - Exclue les `index` de df_duplication
                - then go next
        - Si test_address_complement = True:
            - Test 1: doublon:
                - Oui: append-> `df_not_duplicate`
                - Non: Pass
                - Exclue les `index` de df_duplication

On peut sauvegarder le df_not_duplicate et le restant en tant que special_treatment
